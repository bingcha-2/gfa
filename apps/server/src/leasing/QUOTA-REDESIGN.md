# 额度计算重构方案(Fraction-Share + 保底 N、超卖按真实卖)

> 状态:**已评估,决策口径已敲定(见 §14)**。目标读者:决策人 / 实现者。
> 一句话:把"反推 token 绝对预算"整套估计器删掉,改成**只信上游给的剩余百分比**,绑定号按主人**份额 `e_i = w_i / D`** 分配,**`D = max(N, Σw)`**(`N` = 保底席位数,`Σw` = 真实卖出份额,窗口开始锁定)。结果:**账号永烧不爆(不会被上游封号),5h/周额度直接等于上游真值;卖 ≤ N 有保底 `1/N` + 预留;卖 > N(超卖)按真实卖的切薄到 `1/Σw`,`Σe≤1` 恒成立 → 永不撞墙;超卖的代价是"每席变薄"不是"被拒"。**
>
> **份额模型(决策1,最终):分母 `D = max(N, Σw@窗口开始)`。** 低于 N 保底 `1/N`(不被在册人数稀释、有预留);高于 N 才按真实卖的 `1/Σw` 走(支持超卖)。**加人下个窗口才生效**(`D` 窗口内锁定,不中途重切老人)。取舍详见 §4.1 / §4.4 / §14。
>
> **归因模型(修 bug):用"分段累加 `T_i`"(§4.2),不用瞬时反推 `o_i=(1−fraction)×u_i/Σu`。** 瞬时式有致命 bug:别人猛用会撑大分母把已拦主人的 `o_i` 稀释 → 错误解封;重启/外部消耗会把账砸给"下一个来用的人"。`T_i` 只增不减,两 bug 同治。**另:`getCardWeight` 的 clamp 改 per-account(§7)。**

## 1. 为什么要改(现状的病根)

现在的额度判断散在 4 个文件、约 2000 行,核心问题:

1. **算不准 —— 因为在"猜"上游额度。** 上游(Claude/Codex/Gemini)从不返回绝对 token 预算,只给一个**粗粒度的剩余百分比**。现在的 `QuotaProfileTracker` 用 `预算 = 已用 / (1-剩余%)` 反推,再做学习/衰减/采样。窗口前期数据不足、上游百分比又是 ~20% 一跳,反推天然不准。
2. **周额度纯靠猜。** 没有显式周上限时,用 `5h上限 × R`,`R` 默认 **3.752**。真实周/5h 比因人而异(3~30),默认值对重度用户严重低估。
3. **超卖。** "检查在租出时、扣减在上报时"之间有飞行窗口;再加 `≥90% 剩余无条件放行`,窗口开头基本零执法。
4. **轮换卡污染绑定号。** 轮换卡会跑到被绑定的号上一起烧额度,且其消耗不被公平检查,导致绑定主人被冤枉地提前掐断。

---

## 2. 核心原则

1. **上游的剩余百分比 `fraction` 是唯一真值,只围绕它做判断。** 不反推、不学习、不存绝对预算。
2. **5h(及周,若上游有)直接读上游对应窗口的剩余%。** Claude/Codex 同时返回 5h + 周(`remote-anthropic/auth/claude-usage.ts`、`remote-codex/auth/codex-usage.ts`);**antigravity 只有 5h、无周**(见 §3.1)。**R 系数彻底删除。**
3. **token 仍然计数**(按模型档位加权),但**只用作"分账比例"和账单**,绝不进"账号还剩多少"的判断。
4. **绑定号按主人份额 `e_i = w_i / D`,`D = max(N, Σw)`(窗口开始锁定)。** `N` = 保底席位数(`salesSeatCapacity`,默认 8),`Σw` = 真实卖出份额。卖 ≤ N → 分母 N、保底 `1/N` + 预留;卖 > N(超卖)→ 分母 Σw、切薄到 `1/Σw`。
5. **超卖随便卖,代价是切薄不是撞墙。** `Σe = Σw/max(N,Σw) ≤ 1` 恒成立 → 账号永远够分、永烧不爆、谁都不撞墙。卖越多每席越薄,但永远有得用。**加人下个窗口生效**(`D` 窗口内锁定,不重切老人)。详见 §4.4、§11、§14。

## 3. 卡类型与处理

代码里按 `assignmentPolicy` / 绑定关系分两类(见 `lease-service.ts:516-523`):

| 类型 | 代码判定 | 选号 | 算不算 fair-share 分账 | 卡级静态限额 / 账单 / usage history |
|---|---|---|---|---|
| **轮换卡** | `isPreferredDynamic` 或无绑定 | 独立池里挑剩余%最高的号,低了换下一个 | **不算**(不进 `fairShareTracker`、不影响任何绑定号 `T_i`) | **照记**(与绑定卡完全一样) |
| **绑定卡(主人)** | `hardPinnedAccountId > 0` | 只用自己绑的号 | **算** —— 多主人共号时按 `T_i` 分账 | **照记** |

> **关键区分(本次细化):轮换卡只是不参与"账号份额分账(fair-share)",其余照常记。** 具体地,`reportResult` 里是 **3 个独立机制**,门控**只**加在中间那个:
> | 机制 | 代码 | 作用 | 轮换卡 |
> |---|---|---|---|
> | `accessKeyStore.recordUsage` | `lease-service.ts:1156` | 卡级计数器 → 静态限额(`bucketLimits`/`weeklyTokenLimit`)+ 账单用量 | **照跑**(本就无门控,不动) |
> | `fairShareTracker.recordUsage` | `lease-service.ts:1169` | 账号份额分账(累加 `T_i`) | **跳过** ← 仅此处加 `hardPinnedAccountId>0` 门控 |
> | `tokenUsageTracker.record` | `lease-service.ts:1188` | 逐次 usage history 明细(可查询/持久化) | **照跑**(不动) |
>
> 所以"轮换卡不计额度"**仅指不进 fair-share 分账**;它的静态限额仍会拦、账单仍会算、usage history 仍可查。

**严格分池(C):轮换卡的候选池排除所有绑定号。** 绑定号只服务自己的主人。

**`preferred-dynamic` 完全按轮换卡处理(本次明确)。** 现状它有"展示绑定号优先 + 动态池兜底"的混合行为(`lease-service.ts:1547-1591`,`hardPinnedAccountId=0`)。新口径:
- **不记 fair-share 分账**——给 `fairShareTracker.recordUsage`(`lease-service.ts:1169`)加"仅 `hardPinnedAccountId>0` 才记"门控;**`accessKeyStore.recordUsage`(静态限额/账单)与 `tokenUsageTracker.record`(usage history)不加门控、对所有卡照记。**
- **候选池排除所有硬绑定号**(严格 C);其"展示优先号"降级为**软偏好**,仅当该号本身在轮换池内(非他人硬绑)才优先,否则正常轮换。
- 目的:杜绝 `preferred-dynamic` 漫游到别人硬绑号上烧账号份额(即 §1 第 4 条的"轮换卡污染绑定号")。**注意:这只杜绝份额污染,不影响轮换卡自己的静态限额/账单/history。**

三家共用同一套代码(`lease-core` + `fair-share-tracker` + `quota-profile-tracker`),各自实例化一份 tracker(`token-server.service.ts:53`、`remote-anthropic/service`、`remote-codex/service`)。重构对三家同时生效,但**信号能力不同**:

## 4. 算法详述

对**每个绑定号**、**每种窗口(5h / 周)**、**每个家族桶**独立计算。

### 4.1 记号
- `fraction` ∈ [0,1]:上游给的该号该窗口剩余%(真值;-1 = 未知)。
- `N`:**该号的"保底席位数"**(per-account,= 销售席位数 `salesSeatCapacity`,后台定,默认 8)。定义"**保底每席 = 1/N**"——只要没超卖,人人至少拿到 `1/N`,没卖出的席位预留。
- `w_i`:主人 i 占几席(`getCardWeight`,按会员等级;独占号给 `w=N`)。
- `Σw`:**本窗口开始时**该号所有绑定主人的权重和(= 真实卖出的总份额)。
- `D`:**分母 = `max(N, Σw)`**,**窗口开始时锁定**,窗口内不变。
  - 卖 ≤ N 人(`Σw ≤ N`)→ `D = N` → 保底 `1/N` + 预留(`1 − Σw/N`)。
  - 卖 > N 人(超卖,`Σw > N`)→ `D = Σw` → 每席自动切薄到 `1/Σw`,`Σe = 1`,账号刚好分完。
- `e_i`:主人 i 的保证份额 = **`w_i / D`**。
- `u_i`:主人 i 自上次快照以来的**新增**加权 token 用量(`weightedCost`,**只当比例**)。每次 fraction 快照归并后清零(见 §4.2)。
- `T_i`:主人 i 本窗口**累计已烧的账号比例**(分段累加,**只增不减**)。拦人真正比较的量,替代初稿那个瞬时反推的 `o_i`。

> **核心不变量(自动成立,无需后台约束):`Σe = Σw / max(N,Σw) ≤ 1` 恒成立 → 账号永远够分 → 任何人都不撞墙、账号永烧不爆(不会被上游封号)。**
> - **超卖 = 卖超 N 人(`Σw > N`),代价是"每席变薄"(`1/N → 1/Σw`),不是"被拒"。** 因 `D` 跟着 `Σw` 涨,`Σe` 钉在 1,账号刚好分完,没人撞墙。这是本设计支持超卖的核心(见 §4.4)。
> - **加人下个窗口才生效:`D` 在窗口开始锁定。** 窗口内新绑的人不改 `D`、不重切老人;到下个 reset(`T_i` 本就清零)才用新的 `Σw` 重算 `D`。**所以老人不会被中途加人突然压线。**
> - **为什么不用纯 `w_i/Σw`(始终按在册人数):** 那样卖 ≤ N 人时也会被在册人数稀释、且中途加人立刻重切老人。用 `max(N,Σw)` + 窗口锁定 → 低于 N 有保底、高于 N 才按真实卖的走、且不中途重切。

> **为什么不用瞬时反推 `o_i = (1−fraction)×u_i/Σu`(初稿做法,已废弃):** "我累计烧了多少"是个只增的量,初稿却用"当前瞬时占比"去倒推它。占比的分母会随**别人**用量变大,于是某人明明没再用,反推出的 `o_i` 却被稀释变小——已被拦的主人被错误"解封"。换成**分段累加** `T_i`(只增不减)后根治(无需 latch)。

### 4.2 判定(分段累加 —— 核心)

**两个时机做两件事:**

**(a) 每次 fraction 快照刷新时(`applyAccountQuotaSnapshot(fraction, resetAt)`)→ 把这一段账号消耗归并进各人 `T_i`:**
```
# —— reset 判定:只认 resetAt 前移,绝不靠「fraction 回升」判 reset(修 §15.1)——
若 resetAt 比上次前移(= 新窗口):
    清零所有 T_i / u_i;lastFraction = 本次 fraction(≈1);重算并锁 D + 参与集合 + 预留(见 §4.4)
    return                                          # 新窗口,不在本拍归并

# —— 同窗口内:lastFraction 是「低水位」,只随真实下降前进,回升一律不抬高 ——
Δ账号 = max(0, lastFraction − fraction)             # fraction 回升(噪声/客户端乱序/退款)→ Δ=0
Δu_i  = 本卡这一段新增加权用量 = 当前 u_i
Δu    = Σ Δu_i(本号所有绑定主人,含 idle=0)
若 Δ账号 > 0 且 Δu > 0:  T_i += Δ账号 × (Δu_i / Δu)  # 这一段谁在用,这一段的账就算谁的
若 Δ账号 > 0 且 Δu = 0:  无人认领 → 按 §6 处理(默认不摊;不砸给"下一个来的人")
若 fraction < lastFraction: lastFraction = fraction  # 仅真实下降才降水位;回升/相等不改(低水位单调不增)
归并后:每张卡 u_i 清零                               # lastFraction 不在这里无条件赋值(这是 §15.1 的根因)
```

**(b) 每次租约前 / 上报后 → 拿 `T_i` 直接判定(不再反推):**
```
e_i = w_i / D                                      # D = max(N, Σw@窗口开始),窗口内锁定
拦人:T_i ≥ e_i  → 拦该主人(他那 w_i/D 份累计烧完了)
展示:自份额剩余 = clamp((e_i − T_i) / e_i, 0, 1)   # 见 §9;e_i≤0 返回 0(空且拦,修 §15.4);T_i>e_i 夹到 0
```
**5h 与周各维护一套 `T_i`、各跑一遍,任一超限即拦。**

> **`T_i` 只增不减 → 不可能被别人稀释、不可能被"解封"。** 这是相对初稿(瞬时反推 `o_i`)唯一也是最关键的算法改动,根治了"晚到重用户把已拦主人放出来"的 bug(§4.1)。

**真·不变量(有意义,非恒等式;作 §13.4 测试):**
```
Σ T_i ≤ 1 − lastFraction(低水位)               # 对齐 §15.1 的低水位口径
       ≤ 1 − min(fraction@本窗口)               # 已认领归因之和 ≤ 账号真实已用(以本窗口最低点计)
```
> 注:这条是 **≤**,不是初稿那个永真的 `Σo_i ≡ 1−fraction`(它两边都由 `Σu` 约掉、恒等、测不出任何东西)。新式里 `ΣT_i` 是一段段真实累加出来的,与 `1−lastFraction` 的**差额**正好暴露"未认领消耗",既能当断言又能当监控信号。
> **口径(修 §15.1):断言对齐「低水位 `lastFraction`」而非「瞬时 `fraction`」。** 因 fraction 可能被噪声/乱序短暂抬回水位之上,此时 `1−fraction < 1−低水位`;用低水位才不会把"账号刚被打破又被噪声抬回"误判成越界。真实窗口内 fraction 单调下降时,低水位 == 当前 fraction,与原口径完全一致。

**账号剩余的拆分:**
```
账号真实剩余 = fraction
  = 在册主人合计剩余 (Σe − Σ T_i)               # 已卖出去、还没用掉的
  + 预留            (1 − Σe) = max(0, 1 − Σw/N) # 仅卖 ≤ N 时 >0;超卖时 Σe=1 → 预留=0
  + 未认领消耗      ((1−fraction) − Σ T_i)      # ≥0,默认锁着(§6)
```

### 4.3 数值示例(全卖满,`N=2`,2 主人平权 `w=1` → `e=50%`)
关键:`T_i` 是**一段段累加**出来的,跟"当下谁在用"无关,所以下面这个"先 O1 用、后 O2 猛用"的序列不会把 O1 解封:

**第 1 段:只有 O1 用** —— 账号从剩 100% 掉到剩 70%(`Δ账号=30%`),这段 `Δu` 全是 O1:
- `T₁ += 30% × 1 = 30%`、`T₂ = 0`;都没到 50%,不拦。

**第 2 段:O2 猛用、O1 不动** —— 账号再从 70% 掉到 40%(`Δ账号=30%`),这段 `Δu` 全是 O2:
- `T₁ += 30% × 0 = 0` → `T₁` **仍是 30%**(没被 O2 稀释!);`T₂ += 30% × 1 = 30%`。
- `ΣT = 60% ≤ 1−fraction = 60%` ✓。
- 对比初稿瞬时式:此刻会算 `o₁ = 60%×u₁/Σu`,O2 的大用量把分母撑大 → `o₁` 掉到 30% 以下 → **错误地把已用更多的 O1 显示成更空、甚至解封**。新式不会。

O1 累计用到 `T₁=50%` 被拦后,**号还剩的部分锁给 O2**,O2 睡觉也保留。

### 4.4 卖几人:`D = max(N, Σw)`,保底 N,超卖按真实卖的走(永不撞墙)

`N` 是**保底线**(每席至少 `1/N`),`Σw` 是**真实卖出**。分母 `D = max(N, Σw)`,窗口开始锁定。三档:

**① 卖不满(`Σw < N`,预留):** 号 `N=8`,只绑 2 席(各 `w=1`)→ `D=8`,`e=1/8=12.5%`:
- 每人拦在自己烧到账号 12.5% 时;剩 75% = 6 个未售席位预留(`1 − 2/8`)。
- 第 3 人加入(下个窗口生效)→ `Σw=3` 仍 ≤ 8 → `D` 还是 8 → `e=1/8` 不变,老人份额不动,新人从预留拿。

**② 刚好卖满(`Σw = N`):** 绑 8 人各 `w=1` → `D=8`,`Σe=1`,账号刚好分完,无预留无超卖。

**③ 超卖(`Σw > N`,你要的):** 同号想卖 12 人 → 直接绑 12 人(各 `w=1`)→ `Σw=12 > N=8` → **`D=Σw=12`**,`e=1/12≈8.3%`:
- `Σe = 12/12 = 1` → **账号刚好分完,谁都不撞墙、烧不爆**;利用率拉满。
- **超卖的代价 = 每席变薄**(12.5% → 8.3%),**不是**"晚到者被拒"。卖越多越薄,但永远有得用。
- 想控制超卖力度 → 后台决定绑几个人(`Σw` 多大);`N` 只管"卖不满时的保底大小"。

**关键:无论哪档,`Σe = Σw/max(N,Σw) ≤ 1` 恒成立 → 账号永远够分、永不撞墙、永烧不爆(不会被上游因烧太猛封号)。** 这就是"保底固定 N、高于 N 按真实卖的走"——`D` 一个公式同时覆盖预留和超卖。

**加人下个窗口才生效(重要):** 中途绑新人**不立刻改 `D`**;到下个窗口 reset(`T_i` 清零)才用新的 `Σw` 重算 `D`。所以**老人窗口内不会被突然压线**;新人当前窗口先用预留/暂不参与,下窗口正式分到 `1/D`。

**窗口开始锁定的不止 `D`,还有「参与卡集合」+「预留额」(修 §15.2,关键):** 只锁 `D` **不能**阻止窗口内新绑的卡去取 `w/D` —— 满号(`Σw_locked=N`、无预留)时新卡照取会让存活卡 `Σe>1`、晚到者被烧到 429 撞墙。所以 reset 时一并锁定:
```
窗口 reset 时算定并持久化(per (account,bucket,window)):
  participants  = 此刻绑定的卡集合
  Σw_locked     = Σ w over participants
  D             = max(N, Σw_locked)
  R0            = max(0, 1 − Σw_locked / D)         # 锁定预留(满号/超卖时 R0=0)
  R_avail       = R0                                # 运行时可领预留(随新卡领取递减)

取号/判定时:
  若卡 ∈ participants:  e_i = w_i / D                # 锁定集合,Σe_locked = Σw_locked/D ≤ 1
  若卡 ∉ participants(窗口内新绑):
      claim = min(w_i / D, R_avail)                 # 只能从预留里领,按绑定顺序先到先得
      首次领取时:R_avail -= claim(持久化)
      e_i = claim                                   # R_avail 耗尽(含满号 R0=0)→ e_i=0 → 本窗口不发卡/血条0
```
于是 `Σe = Σe_locked + Σe_new ≤ Σw_locked/D + R0 = 1` **恒成立**,满号加人不会超卖撞墙;新卡到下个 reset 进 `participants` 后正式分 `w/D`。

**单主独占整号** → 给那张卡 `w=N`(`Σw=N` → `D=N`,`e=1`),`T = 1−fraction` 用到账号空。

### 4.5 计算与持久化机制(谁在用 / 并发 / 重启 / 何时回传)

**谁在用 = 逐卡精确记账,不是猜。** 每条上报都带 `cardId`(= accessKey 记录 id),`reportResult → fairShareTracker.recordUsage(accountId, cardId, …)` 把这次的加权 token 累加到**这张卡的段内计数器** `perCard[cardId]`(= `u_i`,"自上次 fraction 快照以来"的增量)。并发多人 = 各自往各自计数器加,互不干扰。**轮换 / `preferred-dynamic` 不进这个 fair-share 计数器**(但它们的卡级静态限额/账单/usage history 照记 —— 见 §3)。

**两步走(per 账号 × 家族桶 × 窗口):**
```
归并(fraction 一刷新就做一次,见 §4.2a 完整版):
  reset?(resetAt 前移)→ 清 T_i/u_i、lastFraction=fresh、重算锁 D;return
  Δ账号 = max(0, lastFraction − fraction)          # 低水位差;回升 → 0
  Δu    = Σ u_i(本号所有绑定主人,含 idle=0)
  T_i  += Δ账号 × u_i/Δu   (Δ账号>0 且 Δu>0 时)
  u_i 清零;仅当 fraction<lastFraction 才 lastFraction=fraction(低水位单调不增,见 §15.1)

判定/血条(随时可算,见 §4.2b):
  拦人 = T_i ≥ e_i
  血条 = clamp((e_i − T_i)/e_i, 0, 1)     # e_i≤0 → 返回 0(空且拦,修 §15.4);T_i>e_i → 夹到 0
```
持久量是 `T_i` + `lastFraction`(都已落库,见 §8),`u_i` 只是两次快照之间的临时增量。**5h 与周各维护一套 `T_i`**,回传 `fairShareQuota`(5h)+ `weeklyFairShareQuota`(周)两组。

**顺序要求(实现注意):** `reportResult` 内必须 **先 `fairShareTracker.recordUsage`(累加 u_i)→ 再 `applyAccountQuotaSnapshot`(刷新 fraction 并触发归并)→ 最后算血条**,否则回传的是上一拍的值。(`accessKeyStore.recordUsage` 静态计数照旧在最前,不受影响。)

**份额=他消耗的,精度边界:** 每一段的 `Δ账号`(上游真值)按**这一段内**各人用量占比 `u_i/Δu` 切分,落进只增的 `T_i`。因严格分池(无外人/轮换),`Δ账号` ≈ 这些主人在这段造成。唯一近似 = "GFA 加权 token ↔ 上游真实计费"的权重映射(来自定价表,偏保守),非"谁在用"。**注意:这只是快照级近似** —— fraction 粗粒度(~20% 跳)/滞后期间,某人可能瞬时略超 `e_i`,下个快照归并时校正(详见 §11)。

**回传时机(决策 B):** **`reportResult` 的响应也带上现算的 `fairShareQuota`/`weeklyFairShareQuota`**(复用取号里的 `getCardQuotaFractions` / `getCardWeeklyQuotaFractions`),客户端每发完一次即更新血条,不必等下次取号。**需客户端读上报响应里的 quota 字段。**(取号响应继续带,作首次/兜底。)

**窗口对齐(回答"账号刷新了这边别还锁着" —— 关键):** fair-share 的窗口**不自己计时,而是对齐上游 reset 边界**。现有 `syncWindow`/`syncWeeklyWindow`(`fair-share-tracker.ts:210/344`)由 `syncFairShareQuotaSnapshot`(`lease-service.ts:955-1003`)用上游快照里的 `hourlyResetAt`/`weeklyResetAt` 喂入 → `windowStart = resetAt − windowMs`。所以 **fraction 和窗口边界来自同一份上游快照**,不会错拍。
- **账号一刷新(上游窗口 reset)**:新快照的 `resetAt` 前移 → `syncWindow` 检测到 `windowStart` 变了 → **必须连 `T_i`、`u_i` 一起清零**,并把 `lastFraction` 设为新快照的 fresh fraction(≈1)。**清零后 `T_i=0 < e_i` → 立刻解锁**,绝不会"账号满了人还被锁"。
- **没有快照喂入时**(纯本地自计时):`ensureWindow` 用 `windowStart+windowMs` 自动到点 reset;因 `windowStart` 当初就对齐过上游,自计时 reset 与上游 reset 基本同刻(±60s 容差)。
- ⚠️ **实现必须改三处的清零**(现在只清 `perCard`):`syncWindow`(`:216`)、`ensureWindow`(`:482`)、`load()` 过期分支(`:537`)都要**额外清 `T_i`**,否则会出现"账号已刷新但 `T_i` 残留 → 误锁"。这是本需求的头号回归点,§13 专门测。

**重启恢复(回答"重启不影响吧"):** `T_i`(累计账)、`u_i`(段内增量)、`windowStart`、`lastFraction` 持久化在 `FairShareWindow`,每 **30s** + **关机** flush,启动 `load()` 读回。优雅重启全恢复;崩溃最多丢 **< 30s** 的段内增量 —— 注意**`T_i` 不丢**(已归并的累计账安全),丢的只是最近一段还没归并的 `u_i`,并入"未认领消耗",由 §6 兜底(**不会**像初稿那样砸给"下一个来取号的人")。窗口已过期(离线期间跨过 reset)则 `T_i`/`u_i` 清零、`lastFraction` 归 1。**不再依赖任何学习数据**(`QuotaProfile` 已删)。

**加权口径(`u_i` 怎么算、数字哪来):** `weightedCost = netInput×w_in + output×w_out + cached×w_cache`,权重 `w` **= 各模型官方每百万 token 单价、归一到 input=1**,源自唯一价目文件 `packages/shared/src/pricing.json`(`pricing.ts` 自动算成 `QUOTA_WEIGHTS` / `CLAUDE_TIER_WEIGHTS`,`product-bucket.ts:71 quotaWeightFor`):
- family(input=1):claude `{1, 5, 0.1}`、gemini `{1, 6, 0.25}`、gpt `{1, 8, 0.1}`(output ≈ 5~8× input、cache ≈ 0.1~0.25× input,**因为官方就这么收**)。
- Claude 分档(以 Opus 输入归一):opus `{1,5,0.1}`、sonnet `{0.6,3,0.06}`、haiku `{0.2,1,0.02}`、fable `{2,10,0.2}`、autocomplete ~`{0.02,…}`(tab 补全直接跳过不计)。

数字"对不对"由上游真实定价保证,**改价只改 `pricing.json` 一处**,权重自动重算。这是用"API 按量价"当"烧了多少订阅额度"的代理(有依据的公开数字)。关键:**权重只影响同号主人间每段的分账比例 `u_i/Δu`,绝不进"账号还剩多少"(那是上游 `fraction` 真值)**;即便代理略偏,也只微调谁分多少、且偏保守(贵模型用户早拦,保护别人),**不会造成超卖**。

## 5. 为什么这套解决了超卖和算不准

| 老问题 | 新方案下 |
|---|---|
| 算不准(反推预算) | **账号永远等于上游真值**;`T_i` 一段段从真实 `Δ账号` 累加而来,不反推、不学习。**前期噪声自动被数学挡住,无需人为门槛。** |
| 周额度靠猜 | **直接读上游周剩余%**,和 5h 一样准。R 系数删除。 |
| 账号被烧穿(被上游封号) | **结构上不可能**:`Σe = Σw/max(N,Σw) ≤ 1`,每段归并 `Σ T_i ≤ 1−fraction ≤ 1`,账号物理上花不超它自己。 |
| 单人独吞 | **不可能**:每人死卡 `w_i/D`(`T_i ≥ e_i` 即拦);`e_i = w_i/max(N,Σw) ≤ 1`,没有一张卡能烧光整号。 |
| 已拦主人被"解封" | **不会**:`T_i` 只增不减,别人猛用不会稀释它(对比初稿瞬时 `o_i` 会被分母 `Σu` 撑大而回退 —— §4.1/§4.3)。 |
| 后到者撞墙(含超卖) | **不会**:`D=max(N,Σw)` → `Σe≤1` → 账号永远够分。超卖只让每席变薄(`1/Σw`),不拒人。 |
| idle 主人被坑 / 中途加人重切 | **不存在**:`D` 窗口内锁定,加人下个窗口才生效;窗口内老人 `e_i` 不变,idle 那份靠"拦活跃超用者"物理留出。 |
| 轮换卡污染绑定号 | **严格分池 + `fairShareTracker.recordUsage` 仅记绑定主人**,轮换/`preferred-dynamic` 进不了绑定号、也不进其 `T_i`(静态限额/账单/history 仍照记)。 |

## 6. 边界与特殊情况

| 情况 | 处理 |
|---|---|
| **某段无人认领**(`Δ账号>0` 但本段 `Δu=0`,如重启丢失最近一段计数 / 账号被 GFA 之外消费) | 这段消耗**不归并进任何 `T_i`**(没有谁的用量可分),留作"未认领消耗"(= `(1−fraction) − Σ T_i`)。**默认不主动按 e_i 分摊**,也**绝不**砸给"下一段第一个来用的人"(初稿瞬时式的致命缺陷:第一个来的人 `u/Σu≈1` → 背全锅)。残口由严格分池(挡外部污染)+ 账号耗尽/429 兜底。升级条款:若"外部直接用绑定号"普遍,把该段按 `e_i` 平摊进各人 `T_i`。 |
| `Δu = 0` 全程(本窗口还没人用) | 没有任何归并,所有 `T_i = 0`,不拦。 |
| `fraction = -1`(上游周缺失,Claude `seven_day` 偶发缺;antigravity 恒无周) | **保留上次好值**(沿用现有 `claude-usage.ts` + 持久层"缺失返回 -1、不覆盖、保留上次好值"的行为);**从未有过值**才让该窗口不参与判断。**不要"视为满 100%"**——那会把真实见底的周限额误放开。注意:fraction=-1 期间不触发归并(`Δ账号` 无定义),`u_i` 继续累积,等下一个有效 fraction 回来一次性归并。 |
| `fraction ≤ 0` / 上游 429 | 账号耗尽,走现有冷却。 |
| 窗口 reset | `T_i`/`u_i` 清零;`lastFraction` 设为新快照 fresh fraction(快照驱动)或 1(自计时/离线过期);窗口对齐上游 reset 时间(`syncWindow`)。**账号一刷新 `T_i` 即清 → 立刻解锁,绝不残留误锁(§4.5)。** |
| 单主独占 | 给 `w=N` → `Σw=N` → `D=N`,`e=1`,`T = 1−fraction`,等价于"用到账号空"。 |
| 主人中途加入(新增绑定) | **当前窗口 `D` 锁定不变 → 老人 `e_i` 不动**;新人当前窗口暂用预留/不参与,**下个窗口** reset 时用新 `Σw` 重算 `D`、新人正式分到 `w/D`。无中途重切、无突兀掐断。 |
| **主人中途解绑/换绑** | 解绑后其 `u_i`/`T_i` 行删除,**不再进 `Δu` 分母**;已烧部分留在 `1−fraction` 里 → 转为"未认领消耗",不重摊给剩余主人;`D` 同样**下个窗口**才用新 `Σw` 重算。 |
| 超卖(`Σw > N`,正常业务) | `D=Σw` → 每席切薄到 `1/Σw`,`Σe=1`,**账号刚好分完、谁都不撞墙**;卖越多越薄。这是支持的正常路径,**不是**异常。 |
| 想控制超卖力度 | 后台决定绑几个人(`Σw`):绑 ≤ N 有保底+预留,绑 > N 按真实卖切薄。`N` 只定"卖不满时的保底大小"。 |
| Gemini/antigravity 20% 粗粒度 | 拦人粒度随之变粗,但与真值一致、偏保守(保护他人),不会不公平。 |
| Claude opus-weekly 子上限 | 现行 `claude-usage.ts` 周只取账号级 `seven_day`,本方案沿用;opus 周子上限由上游 429 兜底(**已知限制**,见 §11)。 |

- **号池 / 轮换 / `preferred-dynamic` 卡(`hardPinnedAccountId=0`,不绑账号)完全不进 `FairShareWindow`** —— check 与 `fairShareTracker.recordUsage` 都被门控挡掉,**不建任何行**。它们的静态限额在 `Subscription.windowState`、history 在 `CardUsageHourly`、账单各自独立,**本次不改动**。
- **本次 schema 净变更**:① `FairShareWindow` 加 `attributedShare`(`T_i`)+ `lockedDenominator`(`D`)两列(可回收 `estimatedBudget`/`confidence` 两条退役列,零新增);② 删 `QuotaProfile` 表。其余表不动。

## 9. 客户端 / 展示

- **血条用"自份额剩余" `clamp((e_i − T_i)/e_i, 0, 1)`**:满条 = 我的份额完好,空 = 我的份额用尽。这与现状语义(`remaining/perCardBudget`)**完全一致**,客户端渲染逻辑不变。
  - **不用** `e_i − T_i`(账号占比):那会让 1/8 的卡满血时血条只到 12.5%,反直觉。
- **绝不显示超卖(硬要求):血条值在服务端就 `clamp(…, 0, 1)` 后再下发**,客户端**再 clamp 一道兜底**(防旧/坏数据),永远渲染 0~100%,**不可能出现 >100% 或负数**。`T_i>e_i`(在飞 margin 瞬时超线)→ 夹到 0(空条),不显示"超了多少"。§13 专测血条边界。
- **"账号空了但我份额还剩"在正常运作下不会发生(数学保证)。** 因为账号只被它的主人用、每人卡在 `e_i`、`Σe ≤ 1` → `Σ T_i ≤ Σe ≤ 1`:卖不满时账号根本空不了(预留碰不到),卖满/超卖时账号见底⟺人人都正好用满、没人有剩。**个人份额没用完,账号总额就不会空。**
  - **唯一例外 = 真·外部消耗**:有人绕过 GFA 直接登录这个账号在外面用,`fraction` 掉了但没记到任何主人头上。这是异常、不是常态。为这个异常兜底,建议**把账号级 `fraction` 作次要指示**(主条仍是自份额),让用户区分"是账号被外部用空了,不是我超了"。
- 字段:**复用现有** `fairShareQuota` / `weeklyFairShareQuota`(数值仍是 0~1 的"我份额剩余"),服务端换算法、语义不变 → 客户端无需改解析。
- **回传时机(决策 B):取号 + 上报响应都带血条。** `reportResult` 响应**新增**回传这两个字段(现算,见 §4.5),客户端每发完一次即刷新血条,不等下次取号。**这一项需客户端改:读上报响应里的 `fairShareQuota`/`weeklyFairShareQuota`。**(旧客户端不读 → 退化为"下次取号才刷新",不影响拦截正确性。)

## 10. 上线 / 迁移

1. **`QuotaProfile` 表/数据可直接丢弃**(无消费方);`estimatedBudget`/`confidence` 两列已退役。
2. **历史 `FairShareWindow` 数据怎么迁(回答"历史数据咋办"):** 老行有 `weightedUsed`(= 旧"本窗口累计用量")+ `lastFraction`,但**没有** `T_i`。上线时**做一次性回填(backfill),不丢历史**:
   - 对每个 `(provider, accountId, bucket)` 组:`Σold = Σ weightedUsed`(组内所有卡);
   - 每张卡 **`T_i ← (1 − lastFraction) × weightedUsed / Σold`**(= 用存档的 fraction 做一次"补归并",把历史已用按旧比例落进累计账;`Σold=0` → `T_i=0`);
   - 然后 **`weightedUsed ← 0`**(段内增量从零开始)、`attributedShare ← T_i`。
   - 这步可在 `load()` 里"懒迁移":读到 `attributedShare` 列不存在/为 null 的老行就地算一次。**已过期窗口**直接 `T_i=0`(本就该重算)。
   - 兜底:即便回填略偏,下一个上游 reset 即把 `T_i` 清零自愈(§4.5),最多影响**当前这一个窗口**,且偏保守(回填的是"已用",不会少算)。
3. **不做双字段过渡**:按 §9,血条语义与现状一致(0~1 自份额剩余),旧 `fairShareQuota`/`weeklyFairShareQuota` 直接承载新值,旧客户端正常显示。
4. **服务端先行,客户端随后跟进展示**(可选)。
5. **冷启动/迁移期接受短暂不严格**:`FairShareWindow` 跨重启沿用、非全冷;无历史时 `T_i` 从 0 起前期少拦,账号耗尽/429 兜底,下个 reset 自愈。**接受**。
6. **灰度按 `provider + accountId`**:对账号集合(或 accountId 哈希百分比)启用新 `checkFairShare`,其余走旧路。对比试点 vs 对照:监控 **`Σ T_i ≤ 1−fraction`(不越界)+ 未认领差额 `(1−fraction)−ΣT_i` + 429/无号率**,跑通再全量;翻 accountId 集合即回滚。

## 11. 取舍与已知限制

- **永不撞墙、永不烧爆账号(自动,无需后台约束):** `D = max(N,Σw)` → `Σe = Σw/D ≤ 1` 恒成立。卖 ≤ N 保底 `1/N`,卖 > N(超卖)切薄到 `1/Σw`,账号都刚好够分。
- **超卖的代价 = 每席变薄,不是被拒。** 同号多卖人 → 每席 `1/Σw` 变小;**永远有得用**,不会"晚到者被拒"。这是有意支持的真·overbooking,且不会赌输(因账号物理分得完)。
- **加人下个窗口才生效 → 老人窗口内不被压线。** `D` 窗口锁定;代价是新人当前窗口要等到下个 reset 才正式分到份额(中间用预留或暂不参与)。
- **份额保证的精度上限 = 快照粒度 + 在飞 margin**:`fraction` 滞后/粗粒度期间,某段 `Δ账号` 滞后到账,主人可能瞬时略超 `e_i`,下个快照归并补上(`T_i` 一次跳到位)。"硬"是**快照级硬**,非逐 token 硬;但 `T_i` 只增不减,**无"被稀释解封"回退**。
- **利用率代价(仅卖 < N 时)**:未售席位份额闲置锁定(预留换保底);超卖时无此代价(`Σe=1` 满载)。
- **opus 周子上限**未单独跟踪,靠上游 429 兜底。
- **Gemini / antigravity 20% 粗粒度 + 无周**:拦截/展示粒度随上游粗;antigravity 仅 5h。
- **未认领消耗**(账号被 GFA 之外消费):默认不预扣,靠 429 兜底;普遍时再切 e_i 分摊(§6)。

## 13. 测试用例(TDD,先写)—— 这是核心功能,必须厚覆盖

> 标 ★ 的是本次新发现 bug 的**回归用例**,缺一不可。分组对应用户六问:A 基础分账、B 窗口对齐/刷新(Q1)、C 重启(Q2)、D 超卖(Q3)、E 轮换卡分账隔离、F 血条/客户端(Q6)、G 历史数据迁移(Q5)、**H 对抗用例(§15 评估新增缺陷的回归 —— A–G 测不出)**、**I 端到端组合场景(超卖 / 多人共用 / 先后时序 / 交叉,#43–70)**。

**A. 基础分账(分段累加)**
1. **单主独占**(`w=N` → `e=1`):用到上游 ~0% 才拦;血条 == 上游剩余%。
2. **双主平权全卖满**(`N=2`):O1 用满 50% 被拦,此时号剩 50% 锁给 O2(**O2 未使用也保留**)。
3. **等级加权**:O1(`w=2`)/O2(`w=1`),`N=3` → 线在 2/3 : 1/3。
4. **真·不变量**:任意时刻 **`Σ T_i ≤ 1 − fraction`**;且每段归并后 `Σ T_i` 的增量 == 本段 `Δ账号`(无未认领时取等)。**不再**断言永真的 `Σo_i ≡ 1−fraction`。
5. **★分段累加 / 不被解封(回归 §4.3)**:序列"O1 先用到 `T₁=30%` → O2 猛用一大笔" → 断言 **`T₁` 仍 == 30%(没被 O2 稀释回退)**、O1 不被解封;`T₂` 按本段独立累加。**初稿会挂、新算法必过的头号用例。**
6. **保底+预留(`Σw < N`)**:`N=8` 只绑 2 人 → `D=8`,每人卡在 12.5%,**剩 75% 预留**;第 3 人下个窗口加入后 `D` 仍 8、`e=1/8`、老人不变。
7. **★超卖(`Σw > N`,你要的)**:`N=8` 绑 12 人(`Σw=12`)→ **`D=12`**,每人 `e=1/12≈8.3%`,`Σe=1`,**12 人都有得用、谁都不撞墙、账号烧不爆**;`Σ T_i ≤ 1` 始终成立。
8. **单人不可独吞**:任一主人累计到 `T_i = e_i = w_i/D` 即被拦;`e_i = w_i/max(N,Σw) ≤ 1`,烧不光整号。
9. **token 仅作比例**:同一段内 `Δu` 比例不变时,绝对 token 量变化不改变 `T_i` 增量/拦截点。
10. **粗粒度 fraction(20% 跳)**:fraction 一次跳 20% → 这一大段按段内 `Δu` 一次性归并,`T_i` 跳到位、不误拦也不漏拦。

**B. 窗口对齐 / 账号刷新(Q1 —— 绝不"账号满了人还锁着")**
11. **★账号刷新即解锁**:O1 `T₁=e₁` 被拦 → 喂入新快照(`hourlyResetAt` 前移、fraction≈1)→ `syncWindow` 触发 → 断言 **`T₁`/`u₁` 清零、`lastFraction` 复位、O1 立刻可取号**。
12. **★三处清零都覆盖**:分别触发 `syncWindow`(快照驱动)、`ensureWindow`(自计时到点)、`load()`(离线跨过 reset)→ 每条都断言 `T_i` 被清(不只 `perCard`)。**防只清 `perCard` 漏清 `T_i` 的回归。**
13. **窗口边界对齐**:`windowStart == resetAt − windowMs`;reset 与上游 reset 同刻(±60s 容差),不早不晚。
14. **5h 与周独立**:5h reset 不动周的 `T_i`;周 reset 不动 5h 的 `T_i`;两套 key(`weeklyBucketKey`)互不串。
15. **周窗口缺失**:`weeklyPercent` 见底→拦(5h 有也拦);上游周**缺失→用上次好值;从未有值→周不参与**(不"视为满")。
16. **fraction=-1 不误归并**:周缺失期间不触发归并(`Δ账号` 无定义),`u_i` 继续累积,下个有效 fraction 回来一次性归并、不丢不重。
17. **★`D` 锁定 / 加人下个窗口才生效**:窗口内绑第 3 人 → 断言 **本窗口 `D`、老人 `e_i` 都不变、老人不被压线**;到下个 reset → `D` 重算为 `max(N, 新Σw)`、新人正式分到 `w/D`。
18. **★`D=max(N,Σw)` 三档**:`Σw<N → D=N`(保底+预留);`Σw=N → D=N`;`Σw>N → D=Σw`(超卖切薄)。各档 `Σe≤1`。

**C. 重启 / 持久化(Q2 —— 重启不影响)**
17. **★优雅重启全恢复**:写入 `T_i`/`u_i`/`windowStart`/`lastFraction`/`lockedDenominator` → flush → 重建 tracker `load()` → 断言全部逐字段还原(含锁定的 `D`),拦截点不变。
18. **★崩溃丢 <30s 只丢 `u_i`、`T_i` 不丢**:归并出 `T_i` 后再积一点 `u_i` 不 flush → 模拟崩溃重启 → 断言 `T_i` 完好,丢的那点 `u_i` 转为未认领差额、**不砸给重启后首个用户**。
19. **离线跨过 reset**:`load()` 时窗口已过期 → `T_i=0`、`lastFraction=1`,不带着旧 `T_i` 误锁。
20. **30s 周期 flush + 关机 flush**:都触发持久化;`dirty` 标志正确。

**D. 超卖 / 烧不爆(Q3 —— 超卖正常路径,账号永不烧爆)**
21. **★账号永不被烧穿**:任意并发序列(含超卖 `Σw>N`)下断言 **`Σ T_i ≤ 1 − fraction ≤ 1`** 恒成立(账号物理花不超自己 → 不被上游封号)。
22. **★`e_i ≤ 1` 自动成立(原"问题①"已消解)**:任意 `w`(哪怕 `w>N`),`e_i = w/max(N,Σw) ≤ 1`,**无需 clamp**;单卡烧不穿。
23. **★超卖优雅(不拒人)**:`N=8` 绑 12(`Σw=12 → Σe=1`)→ 12 人都能用、**无 429/无撞墙**,只是每席 8.3%;验证"超卖是正常路径,代价是切薄"。
24. **在飞 margin**:check 通过后一笔大用量把 `T_i` 冲过 `e_i` → 下个快照归并校正、后续被拦;期间 `Σ T_i` 仍 ≤ 1(不超账号)。

**E. 轮换卡分账隔离(但其余照记)**
25. **严格 C + preferred-dynamic**:轮换 / `preferred-dynamic` 候选池不含任何硬绑定号;独立池全空才报无号;其用量**不**进任何绑定号 `T_i`/`u_i`。
26. **★轮换卡仍记静态/账单/history**:轮换卡上报后,断言 **`accessKeyStore` 卡级计数(静态限额)、账单用量、`tokenUsageTracker` history 三者都照常写入**;只有 `fairShareTracker.recordUsage` 被跳过。

**F. 血条 / 客户端(Q6 —— 绝不显示超卖)**
27. **血条口径**:下发值 == `clamp((e_i−T_i)/e_i, 0, 1)`;`e_i≤0` 返回 0(空且拦,修 §15.4);`T_i>e_i` 夹到 0。
28. **★永不超卖显示**:构造 `T_i>e_i`(在飞 margin)、构造"外部消耗致账号空但我份额还剩"(异常场景)→ 断言下发值始终 ∈ [0,1]、**永不 >1 也不为负**;账号被外部用空时主条不显示超卖,次要指示给账号级 `fraction`。
29. **回传双通道(决策 B)**:取号响应 + `reportResult` 响应都带 `fairShareQuota`/`weeklyFairShareQuota`;旧客户端不读上报响应 → 退化为下次取号才刷新,**拦截正确性不变**。

**G. 历史数据迁移(Q5)**
30. **★backfill 不丢历史**:构造老 `FairShareWindow`(有 `weightedUsed`+`lastFraction`、无 `attributedShare`)→ `load()` 懒迁移 → 断言 `T_i == (1−lastFraction)×weightedUsed/Σold`、`weightedUsed` 归零;`Σold=0` → `T_i=0`;过期窗口 → `T_i=0`。
31. **迁移后自愈**:回填略偏也只影响当前窗口,下个 reset 清零归零(衔接 B11)。

**H. 对抗用例(§15 缺陷回归 —— 当前用例集测不出,缺一不可)**
32. **★fraction 回升/乱序不重复计数(§15.1)**:同一窗口内喂入 fraction 序列 `0.8 → 0.4 → 0.6(stale/噪声,resetAt 未变)→ 0.4` → 断言 **`Σ T_i ≤ 1 − fraction` 始终成立**、`Δ账号` 只认 `resetAt` 变化触发的 reset、回升段 `Δ账号=0` 且 **`lastFraction` 不被抬高**(只信下降)→ 最终 `T_i` 不超过真实消耗。**初稿 `max(0,last−cur)+无条件更新 lastFraction` 会重复计数、此用例必挂。**
33. **★粗粒度 20% 抖动稳定**:fraction 在 `0.4↔0.6` 桶边界反复抖 5 次(resetAt 不变)→ 断言 `Σ T_i` 单调不减且 `≤ 1−fraction_min`,无误拦。
34. **★reset 只认 resetAt(不认 fraction 跳变)**:fraction 从 `0.1` 跳到 `0.95` 但 `resetAt` 未前移 → 断言**不当作 reset**(`T_i`/`u_i` 不清零、`Δ账号=0`);只有 `resetAt` 前移才清零(衔接 B11/B12)。
35. **★满号窗口内加人不超卖撞墙(§15.2)**:`Σw=N`(`D=N`、`Σe=1`、无预留)→ 窗口内绑第 `N+1` 人 → 断言 **新人当前窗口拿不到份额(只能吃 `1−Σe_locked=0` 的预留 → 直接判无份额/不发卡或退化为不参与)**、**存活卡 `Σe ≤ 1` 仍成立**、老人不被压线;到下个 reset 才用新 `Σw` 重算 `D`、新人正式参与。
36. **★欠卖窗口内加人只吃预留**:`Σw=2 < N=8`(`Σe=2/8`,预留 6/8)→ 窗口内绑第 3 人 → 断言新人 `e_3=w_3/D=1/8` 来自预留、`Σe` 仍 `≤1`、老人 `e_i` 不变(与 #35 对照:有预留才允许窗口内参与)。
37. **★`D` 单一真值、懒建行不重算(§15.3)**:窗口内新卡首次 `recordUsage` 懒建行 → 断言新行 `lockedDenominator` **继承窗口锁定值**,不因 `Σw` 含新卡而重算;同一 `(account,bucket,window)` 所有行 `D` 一致。
38. **★reset 时即算 `D`(无人上报也要算)**:窗口 reset 触发但本窗口尚无任何上报 → 断言 `D=max(N,Σw)` 在 reset 钩子里用 `getRecordsBoundTo` 算定并持久化,而非等首次 `recordUsage` 懒建。

## 15. 评估发现的缺陷与修正(本轮新增,落地前必处理)

> 对 §1–§14 的设计做了一轮独立评估,发现 7 处问题。**#1、#2 会导致线上真实错误**(腐蚀核心不变量 / 证伪"永不撞墙"),其余为实现层与边界正确性。每条给出**根因 + 修正 + 对应测试**(见 §13 H 组)。原 §13 A–G 组**测不出 #1、#2**。

### 15.1【高危】fraction 非单调 → `Δ账号 = max(0, last − fraction)` 重复计数(测试 #32–34)
- **根因:** §4.2a 用 `max(0, last − cur)` 兼任"窗口 reset 检测",且归并后**无条件** `lastFraction = cur`(即使 `cur > last`)。上游 fraction 回升时(antigravity 客户端上报的 Google 快照**乱序晚到** / 粗粒度 20% 桶边界抖动 / 上游退款重算),`lastFraction` 被抬高,随后的真实下降被**算两遍**。
- **后果:** `Σ T_i` 超过真实 `1 − fraction` → **打破 §4.2 / §13.4 / §13.21 当硬断言用的 `Σ T_i ≤ 1 − fraction`**,并**提前误拦**老实用户。是会让招牌不变量测试本身 flaky 的 bug。
- **修正(已写进 §4.2a 真值):** `lastFraction` 改成窗口内**单调不增的低水位**,只在 `fraction < lastFraction` 时下降、回升一律不抬;`Δ账号 = max(0, 低水位 − fraction)`;**reset 只认 `resetAt` 前移**(`syncWindow` 已有该信号),与 fraction 是否回升彻底解耦。`Σ T_i` 不变量口径同步改为对齐低水位(见 §4.2「真·不变量」)。
- **改动点:** `applyAccountQuotaSnapshot(fraction, resetAt)` 加 `resetAt` 入参;归并按 §4.2a 新版;`syncFairShareQuotaSnapshot`(`lease-service.ts:955-1003`)把上游 `hourlyResetAt`/`weeklyResetAt` 一并喂入。

### 15.2【高危】窗口内加人到"已卖满/超卖"号 → `Σe ≤ 1` 不再成立、晚到者撞墙(测试 #35–36)
- **根因:** `D = max(N, Σw@窗口开始)` 锁定只阻止 `D` **变大**;但 `e_i = w_i/D` 是**取号时按当前在册卡现算**的。窗口内新绑的卡照样取 `e_new = w_new/D_locked`。号已 `Σw=N`(无预留)时再绑第 `N+1` 人 → 存活卡 `Σe = (N+1)/N > 1`。
- **后果:** 账号可能在所有人到 `e_i` 前被烧到 `fraction≤0` → **晚到者 429 撞墙**,直接证伪"永不撞墙"。§4.4/§6 的文字说新人"暂不参与",但**锁 `D` 不等于排除新卡**,缺实现机制。
- **修正(已写进 §4.4 真值):** reset 时连同 `D` 一起锁定 `participants`(参与卡集合)、`Σw_locked`、预留 `R0 = max(0, 1−Σw_locked/D)` 与运行时 `R_avail`。`卡 ∈ participants → e_i=w_i/D`;`卡 ∉ participants(窗口内新绑)→ e_i = min(w_i/D, R_avail)`,首次领取扣减 `R_avail`,耗尽(满号 R0=0)即 `e_i=0`、本窗口不发卡。保证 `Σe = Σw_locked/D + R0 = 1` 恒成立。
- **改动点:** §4.4 已给伪码;`checkFairShare`/`getCardQuotaFractions` 对"是否 ∈ participants"分两支;`FairShareWindow` 锁定记录加 `R_avail`(随 `lockedDenominator` 同处存,见 #3);新卡首次领取需在判定路径里原子扣减并持久化 `R_avail`。

### 15.3【中危】`lockedDenominator` 按行冗余存 + 懒建行 → `D` 被算花(测试 #37–38)
- **根因:** §8 允许 `D` "冗余写每行"。窗口内新卡首次 `recordUsage` 懒建行(见 `fair-share-tracker.ts:459 getOrCreate` / `:177 recordUsage`),此刻现算 `max(N,Σw)` 会**含新卡** → 新行 `D` 与老行不一致,`Σe` 数学崩。且 reset 时若无人上报,无人触发 `D` 计算。
- **修正:** `D`(+ `participants`/`Σw_locked`/`R0`/`R_avail`)作 per-`(account,bucket,window)` **单一真值**,在**窗口 reset 钩子**里算定持久化;窗口内所有卡只读、新卡继承,绝不重算:
```
// BucketTracker 上挂一份锁定态(非 per-card):
interface LockedShare { D: number; participants: Set<cardId>; sumW: number; reserve0: number; reserveAvail: number; }

onWindowReset(account, bucket):            // resetAt 前移触发(§4.2a),无人上报也要跑
    owners = accessKeyStore.getRecordsBoundTo(account)   // 反向索引,需新建
    sumW   = Σ getCardWeight(o, account) for o in owners // 仅 hardPinnedAccountId>0
    D      = max(N, sumW)
    locked = { D, participants: new Set(owners), sumW, reserve0: max(0,1−sumW/D), reserveAvail: same }
    persist(locked)                        // 单条/小表,不冗余进每张卡的行

eForCard(card):                            // 判定/血条统一走这里
    if card ∈ locked.participants: return w(card)/locked.D
    claim = min(w(card)/locked.D, locked.reserveAvail)
    if 首次参与: locked.reserveAvail -= claim; persist(locked)
    return claim                           // reserveAvail 耗尽 → 0
```
- **改动点:** §8 `lockedDenominator` 由 per-row 改为 per-tracker 单值并加 `reserveAvail`;新增 reset 钩子。**依赖 `getRecordsBoundTo` 反向索引(`access-key-store` grep `:536` 未命中,需新建 —— 见 §7)。**

### 15.4【中危】`w_i=0`/`e_i=0` 时血条满但永远取不到号(测试 #39)
- **根因:** 拦人 `T_i ≥ e_i`:`e_i=0` → `0≥0` 恒真 → 零用量也拦;血条 §9 规定 `e_i=0` 返回**满**。两者矛盾:显示 100% 却租不到号。`getCardWeight` 对未知/异常卡返回 0 是现实兜底值。
- **修正:** `e_i=0` 统一按"**空且拦**"——§9「`e_i=0` 返回满防除零」改为「`e_i=0` 返回 0」;血条与拦截共用一个判定:
```
bloodBar(e_i, T_i):
    if e_i <= 0: return 0          // 无份额 = 空条 + 必拦(原来返回「满」会自相矛盾)
    return clamp((e_i − T_i)/e_i, 0, 1)
allowed = (e_i > 0) && (T_i < e_i) // e_i=0 → 必 false,与血条 0 一致
```
另在选号层对有效硬绑主人保证 `getCardWeight ≥ 1`,把 `w=0` 收敛为"未绑/不发卡",避免出现 `e_i=0` 的活跃卡。

### 15.5【中低】"账号空但我份额还剩"不止外部消耗一个例外(测试 #40)
- **根因:** §9 称唯一例外是"真·外部消耗"。但 §4.5 自承重启丢 `<30s` 的 `u_i` → 转未认领消耗:账号被烧但没记进任何 `T_i`。低用量主人 `T_i<e_i`(血条 >0)仍可能因未认领消耗 + 他人推到账号见底而 429。**内部路径**也能到同一状态。
- **修正:** 仅文档口径修正 —— §9/§11 把"唯一例外"改为"主要来自外部消耗,亦含重启丢失的未认领段(有界 `<30s`,429 兜底)";血条次要指示(账号级 `fraction`)的建议保留。

### 15.6【低】antigravity per-model fraction 与 per-family bucket 粒度错配(测试 #41)
- **根因:** 归并假设一个 `(account,bucket,window)` 对一个 fraction;antigravity 是 `modelQuotaFractions`(按模型)。若一 bucket 含多模型且 fraction 不同,`Δ账号` 取值有歧义。
- **修正:** 确认 bucket 是否细到 per-model;若不是,文档标注为**已知近似**(偏保守取较低 fraction),并在 §3.1 注明。

### 15.7【低】backfill 的 `Σold` 含旧轮换/`preferred-dynamic` 行(测试 #42)
- **根因:** 现 `recordUsage`(`fair-share-tracker.ts:177`)**无门控**,历史 `FairShareWindow` 混有轮换卡用量;§10 的 `Σold` 被污染 → 少算真实主人。
